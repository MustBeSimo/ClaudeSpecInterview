The SPEC Workflow

Transform vague ideas into detailed specifications through structured dialogue with Claude Code. Powered by AskUserQuestionTool.

Why It Matters

Building without a specification is building with hope. Assumptions stay hidden until rework reveals them. The SPEC workflow forces clarity before implementation by having Claude ask the hard questions you didn't know to ask yourself.

The Process

Create

Write a minimal SPEC.md file. 2–5 paragraphs only. Include: core intent, user context, rough scope, key concerns. Don't pretend completeness.

Interview via AskUserQuestionTool

This is where the magic happens. Claude uses AskUserQuestionTool to ask targeted, revealing questions about your spec. Not surface-level "gotchas"—questions designed to expose hidden assumptions, tradeoffs, and gaps you didn't see.

Questions like:

  • "When you say 'real-time', what's the maximum acceptable latency?"
  • "How do you handle conflicts when two users edit simultaneously?"
  • "What's your definition of 'scalable'—users, requests per second, or data volume?"
  • "If this feature fails, what's the business impact?"
"read this @SPEC.md and interview me using AskUserQuestionTool across 2-3 rounds. Each round: ask 2-4 multi-part questions probing technical implementation, data models, failure scenarios, scale requirements, UI/UX tradeoffs, and architectural choices. Make questions non-obvious and build on previous answers. After all rounds, write the refined spec to @SPEC.md with a 'Decisions Made' section."
The Key: AskUserQuestionTool forces you to articulate what you actually mean. "Collaborative" becomes explicit. "Fast" gets measured. "Scalable" gets defined. Finalize & Implement

Claude writes the detailed specification to SPEC.md with explicit decisions, tradeoffs, and architecture. Then: start a fresh Claude Code session with this refined spec to implement. The separation prevents context contamination and gives Claude focused scope for clean code.

The Outcome

Precision in intent precedes elegance in implementation. The specification isn't a constraint—it's a blueprint for inevitability.

The Secret Sauce: AskUserQuestionTool

This entire workflow is powered by Claude Code's AskUserQuestionTool feature. It's what transforms a vague idea into a precise specification through structured, targeted questioning.

Structured Decision Points

AskUserQuestionTool presents 2-4 options per question, creating structured decision points rather than open-ended dialogue. Each answer is explicit and traceable. No ambiguity. No "I think I said that" moments.

Reveals Hidden Assumptions

You think your spec is clear. It's not. The tool asks questions that expose what you didn't consciously define: latency targets, failure scenarios, edge cases, scale thresholds, and priority conflicts.

Questions You Wouldn't Ask

Claude knows which questions matter. "What happens when X fails?" "How do you define success for this feature?" "When does performance matter most?" These aren't obvious—Claude finds them based on domain patterns.

Forces Articulation

Vague thinking cannot survive structured questioning. When asked "What do you mean by 'fast'?", you're forced to say "sub-100ms response time" or "loads in under 2 seconds"—precision emerges through choice, not discussion.

Multi-Round Interviews

Effective spec interviews use 2-3 strategic rounds of questions, each building on previous answers. Early rounds explore scope and constraints. Later rounds probe edge cases and architectural implications. Each round typically asks 2-4 multi-part questions.

Documents Every Decision

The dialogue is the specification. Every question asked, every option chosen, becomes part of your final SPEC.md. The "Decisions Made" section documents architectural choices, tradeoffs accepted, and constraints acknowledged. Nothing is lost—all reasoning is captured.

Why This Matters: Without AskUserQuestionTool, specifications are just wishful thinking written down. With it, they're tested, challenged, and refined before code touches your idea. That's where the weeks of rework get prevented.

Real-World Examples

The SPEC workflow is most valuable for features with 2+ architectural decisions, integration points, or complex business logic. Here's how it prevents rework across different project types:

SaaS Product

Building a Task Dashboard

A team wanted to build a "collaborative task dashboard" but had conflicting visions about priorities, notifications, and team permissions.

The Problem: "Collaborative" meant different things to each stakeholder. Real-time collaboration? Comment threads? Role-based filtering?
Before SPEC
3 months in, rebuilding core features due to conflicting assumptions
After SPEC
All assumptions explicit. Dev started confident with clear architecture
API Integration

Third-party Payment Gateway

Integrating Stripe seemed straightforward until questions arose: should we retry failed transactions? How do we handle disputes? What currencies?

The Problem: Assumed "integration" was just API calls. Ignored critical business logic around edge cases and compliance.
Before SPEC
Launched without retry logic. Lost revenue from recoverable failures
After SPEC
Strategy defined for retries, edge cases, and compliance upfront
Mobile App

Offline-First Social Feed

Wanted an app that works without internet, but the specification didn't address sync conflicts, data consistency, or storage limits.

The Problem: "Offline-first" is an architectural choice with deep implications. Can't be decided mid-implementation.
Before SPEC
Architecture pivoted halfway through after discovering sync complexity
After SPEC
Conflict resolution strategy determined before a single line of code
Admin Tool

Data Export System

Building a tool to "export user data in multiple formats" raised questions: which formats? CSV integrity? Real-time or scheduled? What about 10GB+ datasets?

The Problem: Scope creep disguised as a simple feature. Different stakeholders expected different capabilities.
Before SPEC
Built for small datasets. Crashed with enterprise customers
After SPEC
Architecture handles scale, formats, and async processing from day one

What Changed

Three patterns across all successful SPEC workflows:

1. Hidden Assumptions Surfaced

Every team assumed differently about scope, priority, and edge cases. The interview forces these out early.

2. Architecture Emerged First

Technical choices became explicit before coding. Real-time vs. eventual consistency? Scale assumptions? Decided upfront.

3. Rework Prevented

The dialogue revealed conflicts that would have surfaced during code review or testing—when rework is most expensive.

4. Stakeholder Alignment

By the time development starts, everyone understands what's being built and why. No surprises during handoff.

When to Use SPEC Workflow

IDEAL FOR SPEC

  • Features with 2+ architectural decisions
  • Integration points (APIs, databases, external services)
  • Complex business logic or edge cases
  • Cross-team coordination needed
  • Performance or scale concerns
  • Long-term feature development (2+ weeks)

LESS NECESSARY FOR

  • Simple CRUD features
  • Single-component UI changes
  • Configuration or bug fixes
  • Strictly defined requirements
  • Features under 1 day to implement
  • Iterative/MVP work with known roadmap

The Cost of Clarity

30 minutes of structured questioning on a complex feature prevents 1-3 weeks of rework. The SPEC workflow isn't about perfection—it's about making assumptions explicit so they can be questioned and resolved early, before code changes everything.
